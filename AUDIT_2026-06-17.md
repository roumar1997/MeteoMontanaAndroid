# Auditoría 2026-06-17 — problemas tipo "UI permite / backend bloquea" y afines

Revisión a fondo tras el bug del chat (la UI dejaba escribir pero el backend
descartaba el push en silencio). 5 frentes: reglas Firestore, desajustes
UI↔backend, seguridad backend, push/paridad iOS-Android, offline/outbox.

Severidad: 🔴 ALTA · 🟠 MEDIA · 🟡 BAJA. Cada punto tiene `archivo:línea` y arreglo.

---

## ⚠️ ANTES DE DESPLEGAR LAS REGLAS FIRESTORE (pendiente de la tarea de hoy)

- 🔴 **La PWA NO llama a `POST /api/chat/start`** (`Desktop/MeteoMontana/js/firebase/messages.js:52`).
  Las apps móviles sí (ya arregladas). En cuanto despliegues las reglas nuevas
  (`conversations create: if false`), **la PWA no podrá iniciar chats nuevos**
  (los existentes siguen). Si usas la PWA para chatear, hay que portar la llamada
  a `/chat/start` antes del primer mensaje **antes** de desplegar las reglas. Si
  la PWA ya no se usa para chat, ignóralo.

## 🔴 ALTA — arreglar pronto

1. **IDOR: marcar notificación leída de otro usuario.**
   `MeteoMontanaAPI api/.../web/NotificationController.java:32` → `NotificationInboxUseCase.java:21`.
   `markRead(id)` no comprueba que la notif sea del que llama → cualquiera marca
   como leída la de otro. Arreglo: pasar `me.uid()` y filtrar por `id AND uid`.
   *(Trivial; impacto bajo en datos pero es un fallo de autorización real.)*

2. **Subida de fotos sin validar magic bytes.**
   `MeteoMontanaAPI .../photos/UploadSchoolPhotoUseCase.java:52` y foto de perfil (`/me/photo`).
   Solo valida `Content-Type` (lo controla el cliente) + 5 MB → se puede subir
   cualquier binario como "imagen". Arreglo: verificar cabecera real (JPEG `FF D8 FF`,
   PNG `89 50 4E 47`, etc.) con lista blanca.

3. **Permiso POST_NOTIFICATIONS posiblemente NUNCA se pide (Android 13+).**
   `app/.../MainActivity.kt:65`. El `registerForActivityResult` se registra dentro
   de `onCreate` tras `super.onCreate` y se lanza en línea — la API exige registrarlo
   antes de STARTED (como campo). Puede tirar `IllegalStateException` o no mostrar el
   diálogo → el usuario nunca concede el permiso → **no llega ninguna push** y nadie
   se entera. **Verificar en un móvil Android 13+.** Arreglo: launcher como campo de
   la Activity; `.launch()` en onCreate.

4. **`/chat/notify` devolvía 200 al rechazar (era el bug original) — ya arreglado hoy,
   pero revisar el patrón hermano:** `/notify` ahora devuelve ok solo en self/sin-token;
   el rechazo de autorización ya no se disfraza. ✔️ (referencia: `ChatPushController.java`).

5. **Admin cree que "PUBLICÓ" una contribución y no se materializó nada.**
   `MeteoMontanaAPI .../contribution/ReviewContributionUseCase.java:148,298,339` (+catches `:287,326,375`).
   Si el bloque/escuela destino no existe o `bloquesJson` está mal, se cae a marcar
   `APPROVED` y 200 igual. `SubmitContributionUseCase.java:67` (auto-aprobado admin)
   hereda el fallo. Arreglo: que `approve` falle (404/422) si no materializa.

6. **Optimismo en UI sin rollback + `catch {}` vacío** (mismo patrón del chat).
   El usuario cree que la acción funcionó y no:
   - Borrar cuenta: `app/.../profile/ProfileViewModel.kt:125` y `iosApp/.../AccountView.swift:87`
     → `signOut()` corre aunque `DELETE /api/me` falle (riesgo de tiendas).
   - Aceptar/rechazar solicitud: `app/.../users/FollowRequestsScreen.kt:79` y `iosApp/.../MyActivityView.swift:171`.
   - Seguir/dejar de seguir: `app/.../users/PublicProfileScreen.kt:110` y `iosApp/.../UsersView.swift:157`.
   - Favorito en detalle: `app/.../detail/SchoolDetailViewModel.kt:356` (`catch (_) {}`).
   - Admin approve/reject quita la card sin confirmar: `app/.../admin/AdminViewModel.kt:90` y `iosApp/.../AdminView.swift:55`.
   - Publicar nota: `app/.../detail/SchoolDetailViewModel.kt:367` y `iosApp/.../SchoolDetailView.swift:144`.
   Patrón correcto a copiar: `SchoolListViewModel.toggleFavorite` (revierte en `.onFailure`).

7. **iOS: el chat no aplica `canWrite`** (hermano del bug original).
   `iosApp/.../ChatView.swift:57`. iOS siempre muestra el input y envía confiando
   solo en el 403 del backend; Android sí bloquea (`ChatViewModel.kt:83`). El botón
   "MENSAJE" de iOS (`UsersView.swift:219`) también se muestra siempre. Arreglo:
   replicar `canWrite` (isPublic || iFollow || theyFollow || hay mensajes).

8. **Android pierde borrados de diario hechos offline.**
   `app/.../profile/JournalEntriesScreen.kt:127`. Al fallar la red no encola nada;
   iOS sí (`AccountView.swift:99`, `JOURNAL_DELETE_ID`). Además el `OutboxFlusher`
   de Android **no maneja `JOURNAL_DELETE_ID`** (`OutboxFlusher.kt:60`) → borraría la
   fila sin ejecutar. Arreglar el flusher primero, luego encolar en el catch.

9. **Firestore: `users/{uid}.isPublic` puede no existir → perfiles "sin configurar"
   se tratan como públicos** para `followers`/`journal` (afecta a la PWA).
   `firestore.rules:113,304`. Un usuario que se loguea pero no reclama username no
   tiene `isPublic` → `isPublic != false` = true. Arreglo: escribir `isPublic:false`
   en el create del doc de login y tratar "ausente" igual en ambas reglas.

10. **Firestore: `public_climbs` create con campos forjables.**
    `firestore.rules:139`. El cliente manda `username`/`displayName`/`photoURL`
    sin validar → suplantar nombre/foto de otro en el feed público (`allow read: if true`).
    Arreglo: no confiar en esos campos del cliente o validarlos.

## 🟠 MEDIA

- **Doble fuente de verdad de `isPublic`**: Postgres (backend/móvil) vs Firestore
  (PWA/reglas) pueden discrepar; un atacante elige el camino más laxo. Unificar/sincronizar.
- **`/api/users/*/journal` y `*/stats` son `permitAll`** (`SecurityConfig.java:64`): el
  diario de un **privado** está bien protegido (404), pero el de los **públicos** es
  accesible **sin login**. Decisión de producto: si quieres exigir login, pasarlos a
  `authenticated()`.
- **Sin rate limiting** en públicos (`/api/schools` recalcula SHA-256 por request,
  `/api/forecast/by-location` proxy a Open-Meteo, búsquedas). Añadir throttling + cachear ETag.
- **`journal` update sin validar campos/tamaños** (`firestore.rules:331`), al contrario
  que el create. Replicar validaciones.
- **Duplicado de NOTE/CONTRIBUTION** si el POST tiene éxito pero el cliente ve timeout
  (`OutboxFlusher.kt:61`, sin idempotencia como sí tiene JOURNAL). Clave de idempotencia.
- **Reintentos infinitos del outbox ante 4xx** (`OutboxFlusher.kt:95`): no distingue
  error de red de error permanente; `retries` se incrementa pero nunca se consulta. Tope/descartar 4xx.
- **Dedupe de "vía hecha" sin sector** (`SchoolDetailViewModel.kt:135` `schoolId|name`):
  dos vías con el mismo nombre en sectores distintos colisionan → ✓ en ambas y se
  pierde una entrada. Incluir sector en la clave.
- **`clearFilters()` incompleto en Android** (`SchoolListViewModel.kt:408`): solo
  resetea distancia, no estilo/roca/texto. iOS sí limpia todo.
- **iOS no encola CONTRIBUTION/NOTE offline** (`ProposeFlow.swift:189`) ni drena el
  outbox al recuperar red (solo en foreground, `MeteoMontanaApp.swift:36`). Android sí.
- **Mover bloque/escuela directo como admin no existe en iOS** (`SchoolDetailView.swift:533`
  siempre usa la cola; Android tiene ruta directa `ProposeContributionFlow.kt:196`).
- **Deep-links de notificación incompletos en iOS** (`NotificationsView.swift:51`): solo
  `follow_request`/`user`/`school`; faltan `chat`/`submission`/`compare`. Relevante al activar APNs.
- **`etagStore.set` corre aunque falle el guardado de caché** (`SchoolListViewModel.kt:199`):
  un 304 futuro serviría caché desincronizada. Guardar etag solo si `replaceAll` tuvo éxito.
- **Token FCM acoplado a `getMyProfile()`** (`AppRoot.kt:51`): si el perfil falla por red,
  el token no se sube esa sesión (catch silencioso). Subirlo independientemente.

## 🟡 BAJA

- `usernames` `read: if true` permite enumerar uid↔username sin login (`firestore.rules:128`).
- `following` create amplio (`firestore.rules:122`): un seguido puede escribir en mi
  subcolección `following`. Acotar a que exista solicitud previa.
- `SavedForecast` crece sin límite y no se purga al des-guardar escuela (`SavedSchoolRepository.kt:67`).
- Notas sin saneo de caracteres de control/HTML (`CreateNoteUseCase.java:35`); el chat sí lo hace.
- Avatar descargado de forma bloqueante en `onMessageReceived` (timeout 4s, `PushService.kt:97`).
- `onNewToken` se traga el fallo sin reintento (`PushService.kt:44`).
- Sin migraciones SQLDelight reales: cambiar schema = bumpear nombre de BD = **se pierde
  el outbox pendiente** (`DatabaseFactory.kt`). Migraciones `.sqm` o drenar antes.

## ℹ️ Conocido / decisión, no bug

- **iOS APNs OFF** (`PushManager.swift:22` `enabled=false`): hoy iOS NO recibe ninguna
  push del sistema (chat, follows, alerta, admin) ni sube token FCM. Solo notifs in-app
  (campana). Requiere cuenta Apple Developer de pago. Ya documentado.
- Storage rules: validan dueño + 5 MB + image/* (falta magic bytes, ver 🔴2).
- Verificado correcto: AdminGuard en todos los `/api/admin/**` y acciones admin; ownership
  en borrar bloque/foto y journal ajeno; DTO público no filtra email/fcmToken/ubicación;
  `/me/**` derivan uid del token (sin IDOR); toggle marcar↔desmarcar offline cancela bien
  el par; idempotencia de JOURNAL en el flush; versión de BD Android == iOS.
